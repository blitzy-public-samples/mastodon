# Streaming Service (`@mastodon/streaming`)

Mastodon's standalone real-time delivery service — the `@mastodon/streaming` package `Source: streaming/package.json:L2`, described in its package metadata as "Mastodon's Streaming Server" `Source: streaming/package.json:L8` — is a separate Node.js process written as ESM (`"type": "module"`, `engines.node ">=22"`) `Source: streaming/package.json:L5-L10`. It pushes timeline and notification updates to clients over WebSocket (via the `ws` `WebSocketServer`) `Source: streaming/index.js:L136` and Server-Sent Events (responses with `Content-Type: text/event-stream`) `Source: streaming/index.js:L931`. It runs independently of the Rails `web` process — its own entrypoint is `node ./index.js` `Source: streaming/package.json:L16`, and its container launches `node ./streaming/index.js` while exposing port `4000` `Source: streaming/Dockerfile:L109-L111`. In production it is reachable only through the nginx `location ^~ /api/v1/streaming` rule `Source: dist/nginx.conf:L126`, which proxies to the `streaming` upstream on `127.0.0.1:4000` `Source: dist/nginx.conf:L10-L15`; the full routing topology lives in [`docs/deployment-topology.md`](../docs/deployment-topology.md).

## Key files

- **`index.js`** — bootstrap/orchestration entrypoint (1,478 lines): loads environment via `dotenv` `Source: streaming/index.js:L26-L34`, then wires a PostgreSQL pool, Prometheus metrics, Redis clients, the HTTP server, and the WebSocket server `Source: streaming/index.js:L128-L136`; it enforces authentication `Source: streaming/index.js:L551`, resolves a channel name to stream IDs `Source: streaming/index.js:L1073`, and forwards Redis pub/sub messages to connected clients — dispatching each inbound channel message through `onRedisMessage` `Source: streaming/index.js:L294-L309`, then on through `streamFrom`/`transmit` `Source: streaming/index.js:L665-L680` to each client's WebSocket `Source: streaming/index.js:L983-L991` or SSE `Source: streaming/index.js:L1027-L1035` output.
- **`database.js`** — turns environment variables into a reusable `pg.Pool` configuration via `configFromEnv` `Source: streaming/database.js:L11` and returns a singleton pool via `getPool`, with optional development-only query logging `Source: streaming/database.js:L115-L132`.
- **`redis.js`** — turns the environment into `ioredis` configuration via `configFromEnv`, supporting direct `REDIS_URL`, Redis Sentinel, or host/port deployments `Source: streaming/redis.js:L66` `Source: streaming/redis.js:L80-L110`, and creates clients with standardized error logging via `createClient` `Source: streaming/redis.js:L118-L127`.
- **`metrics.js`** — sets up the `prom-client` registry and default Node.js process metrics `Source: streaming/metrics.js:L21-L23`, seeds gauges and counters for connected clients/channels, Redis activity, and PostgreSQL pool state `Source: streaming/metrics.js:L25-L75`, and exposes the request handler `Source: streaming/metrics.js:L98-L108` that `index.js` mounts as the Express `/metrics` endpoint `Source: streaming/index.js:L268`.

Supporting modules:

- **`logging.js`** — centralizes structured Pino logging `Source: streaming/logging.js:L39` with per-request IDs surfaced as an `X-Request-Id` header `Source: streaming/logging.js:L11-L24` and an environment-driven log level `Source: streaming/logging.js:L106-L114`.
- **`errors.js`** — defines the `RequestError`/`AuthenticationError` types and a normalizer that maps trusted exceptions to `{ statusCode, errorMessage }` `Source: streaming/errors.js:L15-L46`.
- **`utils.js`** — shared environment-parsing and normalization helpers: `isTruthy` `Source: streaming/utils.js:L23`, `normalizeHashtag` `Source: streaming/utils.js:L49`, `firstParam` `Source: streaming/utils.js:L57`, and `parseIntFromEnvValue` `Source: streaming/utils.js:L73`.

## Conventions

- **Standalone ESM Node service, decoupled from the Rails monorepo deployment.** The package is `private`, ESM (`"type": "module"`), and pins `engines.node ">=22"` `Source: streaming/package.json:L5-L10` and its package manager to `yarn@4.17.0` `Source: streaming/package.json:L4`; it starts via `scripts.start` = `node ./index.js` and type-checks via `scripts.typecheck` = `tsc --noEmit` `Source: streaming/package.json:L16-L17`, and its container runs `node ./streaming/index.js` `Source: streaming/Dockerfile:L111`. **Why:** real-time fan-out scales independently of the request tier, and Node's event-loop model suits many long-lived connections far better than synchronous Rails request workers.
- **Redis pub/sub is the bridge between Rails and streaming.** The service opens a dedicated Redis subscribe client `Source: streaming/index.js:L256` and dispatches each inbound channel message to connected clients via `onRedisMessage` `Source: streaming/index.js:L294-L309` rather than calling Rails directly. **Why:** Rails and Sidekiq publish events onto Redis channels — e.g. `PushUpdateWorker` calls `redis.publish(@timeline_id, message)` `Source: app/workers/push_update_worker.rb:L32-L33` — and streaming merely subscribes, which avoids any direct Rails↔streaming coupling.
- **nginx must give this service a dedicated, unbuffered, upgrade-capable location.** The `^~ /api/v1/streaming` block `Source: dist/nginx.conf:L126` disables response buffering (`proxy_buffering off`) `Source: dist/nginx.conf:L134` and forwards the `Upgrade`/`Connection` headers `Source: dist/nginx.conf:L137-L138`, in contrast to the buffered web path (`proxy_buffering on`) `Source: dist/nginx.conf:L152`. **Why:** WebSocket and SSE need a persistent, unbuffered, low-latency connection, so buffering responses would break real-time delivery (detailed in [`docs/deployment-topology.md`](../docs/deployment-topology.md)).
