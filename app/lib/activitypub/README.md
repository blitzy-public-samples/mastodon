# app/lib/activitypub

The **ActivityPub federation protocol** implementation. Top-level files coordinate inbound dispatch and outbound serialization; the `activity/` subfolder holds per-verb inbound activity handlers (e.g., `Source: app/lib/activitypub/activity/create.rb:3`) and the `parser/` subfolder holds inbound object parsers (e.g., `Source: app/lib/activitypub/parser/status_parser.rb:3`). This directory works alongside the sibling directories that complete the federation picture: `app/controllers/activitypub` (inbound HTTP endpoints, e.g., `Source: app/controllers/activitypub/base_controller.rb:3`), `app/serializers/activitypub` (outbound document shapes, e.g., `Source: app/serializers/activitypub/note_serializer.rb:3`), and `app/workers/activitypub` (async delivery, e.g., `Source: app/workers/activitypub/delivery_worker.rb:3`).

## Conventions

- **Inbound activities are dispatched to per-verb handler subclasses.** `ActivityPub::Activity.factory` calls `klass_for(json)`, a case statement that maps the AP `type` (`Create`, `Announce`, `Delete`, `Follow`, `Like`, `Block`, `Update`, `Undo`, `Accept`, `Reject`, `Flag`, `Add`, `Remove`, `Move`, `QuoteRequest`, `FeatureRequest`) to a handler class in the `activity/` subfolder (`create.rb`, `announce.rb`, `follow.rb`, …); the base `#perform` raises `NotImplementedError`, so a subclass must override it. **Why:** one focused handler per activity verb keeps protocol handling isolated and independently testable. `Source: app/lib/activitypub/activity.rb:19-20`, `Source: app/lib/activitypub/activity.rb:24-25`, `Source: app/lib/activitypub/activity.rb:30-63`
- **Outbound objects are JSON-LD serialized through the adapter + serializer pair.** `ActivityPub::Adapter` (a subclass of `ActiveModelSerializers::Adapter::Base`) builds the `@context` and merges it into the serialized hash, while `ActivityPub::Serializer` (a subclass of `ActiveModel::Serializer`) declares the named contexts (`self.context`) and context extensions (`self.context_extensions`). **Why:** federated peers require valid JSON-LD documents that carry an `@context`. `Source: app/lib/activitypub/adapter.rb:3`, `Source: app/lib/activitypub/adapter.rb:23`, `Source: app/lib/activitypub/serializer.rb:3`, `Source: app/lib/activitypub/serializer.rb:19`, `Source: app/lib/activitypub/serializer.rb:25`
- **Federated requests are authenticated by HTTP Signatures and Linked-Data signatures.** `ActivityPub::LinkedDataSignature` verifies (`verify_actor!`) and signs (`sign!`) the JSON-LD payload, while inbound controllers `include SignatureVerification` for HTTP Signature checks. **Why:** federation is trustless server-to-server between independent instances, so a request's origin must be cryptographically proven. `Source: app/lib/activitypub/linked_data_signature.rb:13`, `Source: app/lib/activitypub/linked_data_signature.rb:36`, `Source: app/controllers/activitypub/base_controller.rb:4`
